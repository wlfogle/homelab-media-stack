## 1. Jellyfin DLNA browse 404 (primary noise source)

```
ERROR (SyncWorker_*) [Jellyfin.jellyfin_apiclient_python.http] 404 Client Error:
  Not Found for url: http://192.168.12.231:8096/Users/<uid>/Items/
    media-source://dlna_dms/hdhomerun_dms_1048eee4/:Channels?Fields=…
jellyfin_apiclient_python.exceptions.HTTPException: (404, HTTPError(...))
```

**Root cause.** The HA Jellyfin integration treats every `media-source://` URI returned by Jellyfin as a browseable `Items/<id>` call. Jellyfin exposes a DLNA DMS entry for the HDHomeRun tuner, but DLNA DMS children are not resolvable through the `Users/<uid>/Items/<id>` API — Jellyfin returns `404`. The exception bubbles up as `Unknown error (unknown_error)` in the websocket browse handler (`homeassistant/components/jellyfin/browse_media.py` → `get_media_info`).

This is the same class of bug tracked upstream in HA core (Jellyfin integration does not filter non-Item media sources). It is cosmetic in that it doesn't break anything else, but every media-browser expand in the HA UI throws it.

**Remediation — pick one (both are non-destructive):**

1. **Hide the DLNA DMS HDHomeRun child in Jellyfin** (keeps the native HDHomeRun tuner):
   - Jellyfin → Dashboard → Live TV → HDHomeRun: keep.
   - Jellyfin → Dashboard → Playback → DLNA → untick "Enable DLNA Server" and "Enable DLNA play-to". This removes the DMS source so HA can't traverse into it.
2. **If DLNA is needed for other clients**: in HA, open Settings → Devices & services → Jellyfin → the server entry → "System options" and uncheck "Enable media browser" for this entry. Playback and presence sensors keep working; the broken browse tree is gone.

**Verification.** After either change, reload the Jellyfin integration (`Settings → Devices & services → Jellyfin → … → Reload`) and expand Media in the HA sidebar; the 404 backtrace should stop appearing.

## 3. IPP integration timeout

```
ERROR [homeassistant.components.ipp.coordinator] Error fetching ipp data:
  Invalid response from API: Timeout occurred while connecting to IPP server.
```

**Root cause.** The configured IPP printer is offline or unreachable on the LAN at the time of polling.

**Remediation.**

- If the printer is intended to be online: power it on, confirm a static DHCP reservation, and verify port 631 from HAOS: `nc -zv <printer-ip> 631`.
- If the printer is retired: Settings → Devices & services → IPP → Delete.
